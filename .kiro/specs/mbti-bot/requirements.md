# Requirements Document

## Introduction

wikiwiki.jp/16types のコンテンツを知識源として、ユーザーが自分のMBTIタイプを診断したり、各タイプ・心理機能・心理傾向軸について詳しく調べることができるNode.js製CLIボットを構築する。Anthropic Claude APIはオプション機能（コンテンツ閲覧時のAI要約）にのみ使用し、診断機能とビルトイン情報閲覧はAPIなしでも動作する。

---

## Requirements

### Requirement 1: 動作モードの定義と起動時判定
**Objective:** ユーザーとして、起動時にどの機能が使えるか即座に分かりたい。そうすることで、設定不足で機能を呼び出して止まることを防げる。

#### Acceptance Criteria
1. The MBTIボット shall 起動時に **ANTHROPIC_API_KEY 環境変数の有無のみ** で **初期モード** を決定する
   - `ANTHROPIC_API_KEY` の値をトリムした結果が **1文字以上** → **APIモード**（全機能利用可）
   - 未設定・空文字列・空白のみ のいずれか → **スタンドアロンモード**（診断・ビルトイン/Web静的表示のみ、AI要約不可）

   **用語定義:** 本仕様では **初期モード** は起動時に ANTHROPIC_API_KEY のみで決まる値を指し、**実効モード** はセッション中の任意時点での実際の動作モードを指す。初期モードが APIモードの場合、API 401 受信により実効モードがスタンドアロンに変わりうる（Req 1.3）。初期モードがスタンドアロンの場合、実効モードは常にスタンドアロンのまま変わらない。テンプレート選択・API呼び出し判定等の全ての条件分岐は **実効モード**（Req 5.3 の `apiMode`）を参照する

2. The MBTIボット shall Web取得の成否は **実効モードの判定に影響しない**（実効モードは初期モードと API 401 のみで決定する）。個別リクエストの通信成否はテンプレート選択（Req 11.3）で処理する。なお、成功済み URL のキャッシュ（Req 8.6）は通信最適化であり、本条件と矛盾しない
3. The MBTIボット shall Claude API呼び出しで **HTTP 401** が返った場合、「APIキーが無効です。ANTHROPIC_API_KEY を確認してください。」と表示し、**そのプロセスが終了するまでスタンドアロンモードへ切り替える**（以後のリクエストで API は呼び出さない）。これが起動後にモードが変わる唯一のケースとする（詳細は Req 9）。**降格後、次回以降の MAIN_MENU 入場時**（降格直後の遷移も含む）には MAIN_MENU メニューの直前に `[モード: スタンドアロン - API 401により切り替え済み]` を1行表示する（毎回表示する。起動時のモード表示 1.4 とは別の文言とする）
4. When 起動する, the MBTIボット shall モード名を1行表示する（例: `[モード: APIモード (claude-sonnet-4-6)]` または `[モード: スタンドアロン - AI要約無効]`）。**この表示は起動時1回のみとし、MAIN_MENU への再入場時には表示しない**。APIモードのバナーに表示するモデル名は **Req 15.1 のデフォルト適用後の実効値**（`ANTHROPIC_MODEL` がトリム後に空なら `claude-sonnet-4-6`、非空ならその値）を使用する。APIモード表示は `ANTHROPIC_API_KEY` が存在することのみを意味し、モデル名が有効であることや API 呼び出しが成功することを保証しない（モデル名の事前バリデーションは行わない。不正なモデル名による API エラーは Req 9.2 の通常エラー処理で対応する）
5. When スタンドアロンモードで起動する, the MBTIボット shall `ANTHROPIC_API_KEY` の設定方法を1行案内する（例: `export ANTHROPIC_API_KEY=sk-ant-...`）。**この案内は起動時1回のみとし、MAIN_MENU への再入場時および API 401 によるモード切り替え後の再入場時には表示しない**
6. The MBTIボット shall 起動時の出力を以下の **固定順序** で行う（各項目間に空行は挿入しない。MAIN_MENU のメニュー表示形式は Req 5.12 内の MAIN_MENU表示形式に従う）

   **APIモード:**
   ```
   [モード: APIモード (claude-sonnet-4-6)]
   =============================
    MBTI Bot
   =============================
   1. MBTI診断
   ...
   ```

   **スタンドアロンモード:**
   ```
   [モード: スタンドアロン - AI要約無効]
   APIキーを設定するとAI要約が利用できます: export ANTHROPIC_API_KEY=sk-ant-...
   =============================
    MBTI Bot
   =============================
   1. MBTI診断
   ...
   ```

   出力順: ① モード表示（1.4）→ ② APIキー案内（スタンドアロン時のみ、1.5）→ ③ MAIN_MENUメニュー。①②は起動時1回のみ、③はMAIN_MENU再入場時にも表示する

7. スタンドアロンモード（起動時設定またはAPI 401切り替え後）でコンテンツを表示する場合、the MBTIボット shall AI要約（テンプレートA）を使用しない。Web取得成功時はテンプレートB、失敗時はテンプレートCで表示する。スタンドアロンモードでAI要約が使えないこと自体は追加通知しない。ただし **Web取得の失敗（404・タイムアウト等）は Req 9.1 に従い通常通りエラーメッセージを表示する**

### Requirement 2: MBTI診断機能（ロジック定義）
**Objective:** ユーザーとして、一貫した基準で同じ回答から同じ結果が得られる診断を受けたい。そうすることで、診断結果の再現性を確認できる。

#### Acceptance Criteria
1. The MBTIボット shall 4軸（E/I・S/N・T/F・J/P）それぞれに **4問**、合計 **16問** の固定質問セットを持つ
   - 各軸4問×±1スコア = 合計値は -4, -2, 0, +2, +4 のいずれか（0の同点が発生しうる）
2. The MBTIボット shall 質問を軸ごとにまとめて出題する（E/I → S/N → T/F → J/P の順に各4問）
3. The MBTIボット shall 各質問に対して `A` または `B` の2択のみで回答させる
4. The MBTIボット shall 回答のスコアリングを以下のルールで行う
   - 各軸の「第1指標（E/S/T/J）寄りの選択肢」を選んだ場合: +1点
   - 各軸の「第2指標（I/N/F/P）寄りの選択肢」を選んだ場合: -1点
   - 各軸のスコア合計が **1以上** → 第1指標を採用
   - 各軸のスコア合計が **-1以下** → 第2指標を採用
   - 各軸のスコア合計が **0（同点）** → E/I軸はI、S/N軸はN、T/F軸はF、J/P軸はP を採用（MBTIの統計的傾向に基づく意図的なデフォルト設定）
5. When 16問全ての回答が完了する, the MBTIボット shall 4軸の判定を組み合わせて16タイプのいずれか1つを確定し、タイプ名・得点内訳を表示する。

   **得点内訳の表示形式:** `[採用指標][軸スコア]` の形式で4軸を `/` 区切りで並べる。軸スコアは **第1指標方向を正**とした符号付き整数（例: S/N軸では S方向が正、N方向が負）。同点時は `[採用指標]0→[採用指標]` と表示する。

   表示例: `E+2 / N-4 / F0→F / P-2`（E+2 = E方向に+2点でE採用、N-4 = N方向に4点なので軸スコアは-4でN採用、F0→F = 同点でFを採用、P-2 = P方向に2点なので軸スコアは-2でP採用）

6. When 診断結果が表示される, the MBTIボット shall ビルトイン概要説明（100字以内）・参考URL を表示し、その直後に Req 12 の免責表示を出力する
7. If ユーザーが質問回答中に `q` または `quit`（大文字小文字不問）を入力する, the MBTIボット shall 「診断を中断しました。」と表示してMAIN_MENUへ遷移する（`0` は診断中は無効）
8. The MBTIボット shall 以下の固定質問セットと採点キーをソースコード内の定数として保持する（内容変更禁止。変更時は全テストを再実施すること）

   | 問 | 軸 | 質問文 | A の意味（スコア） | B の意味（スコア） |
   |---|---|---|---|---|
   | Q1 | E/I | 新しい環境に入ったとき、あなたはどちらですか？ | 積極的に周囲に話しかける（E / +1） | まず周囲の様子を観察する（I / -1） |
   | Q2 | E/I | 週末の過ごし方として自然なのはどちらですか？ | 友人と外出して活動する（E / +1） | 家でゆっくり一人で過ごす（I / -1） |
   | Q3 | E/I | グループ討論でのあなたのスタイルはどちらですか？ | 積極的に発言してリードする（E / +1） | 考えをまとめてから発言する（I / -1） |
   | Q4 | E/I | エネルギーが回復するのはどちらですか？ | 人と会って会話や活動をしたとき（E / +1） | 一人で静かな時間を過ごしたとき（I / -1） |
   | Q5 | S/N | 問題を解決するとき、あなたはどちらですか？ | 過去の実績や具体的な方法を参考にする（S / +1） | 直感やアイデアで新しい方法を試みる（N / -1） |
   | Q6 | S/N | 情報を集めるとき、重視するのはどちらですか？ | 具体的な事実やデータ（S / +1） | 全体的なパターンや意味（N / -1） |
   | Q7 | S/N | 日常生活でより意識するのはどちらですか？ | 現在の具体的な出来事や状況（S / +1） | 将来の可能性や抽象的なアイデア（N / -1） |
   | Q8 | S/N | 趣味や読書の好みはどちらですか？ | 実用的・現実的な内容（S / +1） | 理論的・哲学的・想像力豊かな内容（N / -1） |
   | Q9 | T/F | 重要な決断をするとき、あなたはどちらですか？ | 論理的な分析と客観的な基準を重視する（T / +1） | 関係する人の気持ちや価値観を重視する（F / -1） |
   | Q10 | T/F | 友人が悩みを相談してきたとき、あなたはどちらですか？ | 原因を分析して解決策を提案する（T / +1） | まず気持ちに共感して話を聞く（F / -1） |
   | Q11 | T/F | 仕事やプロジェクトで何を評価しますか？ | 効率・成果・公平性（T / +1） | チームの士気・調和・個人への配慮（F / -1） |
   | Q12 | T/F | 意見の対立が起きたとき、あなたはどちらですか？ | 論理的に正しい側が優先されるべき（T / +1） | お互いの気持ちを尊重して妥協点を探す（F / -1） |
   | Q13 | J/P | 予定の立て方はどちらですか？ | 事前にしっかり計画を立て、その通りに実行したい（J / +1） | 流れに任せて柔軟に対応したい（P / -1） |
   | Q14 | J/P | 締め切りへの対応はどちらですか？ | 余裕を持って早めに終わらせる（J / +1） | 締め切り直前に集中して仕上げる（P / -1） |
   | Q15 | J/P | 生活空間や作業環境はどちらですか？ | 整理整頓されていないと落ち着かない（J / +1） | 多少散らかっていても気にしない（P / -1） |
   | Q16 | J/P | 仕事の進め方はどちらですか？ | 一つのことを完結させてから次に進む（J / +1） | 複数のことを同時並行で進める（P / -1） |

### Requirement 3: MBTIタイプ詳細情報閲覧機能
**Objective:** ユーザーとして、特定タイプの情報をカテゴリ別に調べたい。そうすることで、自分や他者の性格タイプを多角的に理解できる。

#### Acceptance Criteria
1. When TYPE_SELECT状態でタイプ名が入力される, the MBTIボット shall Req 6 の正規化ルールを適用した後 TYPE_CATEGORY状態へ遷移しカテゴリメニューを表示する
2. The MBTIボット shall 以下の **カテゴリ正本名テーブル** を唯一の正とし、UI表示（メニュー・見出し）と Claude プロンプト（Req 15.2）の両方で同じ名前を使用する。URLスラッグ（Req 7.1）は URL構築専用であり、UI名・プロンプト名とは別物とする

   | カテゴリ番号 | カテゴリ正本名 |
   |---|---|
   | 1 | 基本的な特徴 |
   | 2 | 美徳と限界・挑戦課題 |
   | 3 | 人間関係・恋愛 |
   | 4 | 趣味 |
   | 5 | ストレスと対処法 |
   | 6 | リーダーシップ |
   | 7 | 適職・キャリア |

   The MBTIボット shall TYPE_CATEGORYメニューを以下の形式で表示する（カテゴリ名はカテゴリ正本名テーブルの値を使用する）
   ```
   === [タイプ名] を調べる ===
   1. 基本的な特徴
   2. 美徳と限界・挑戦課題
   3. 人間関係・恋愛
   4. 趣味
   5. ストレスと対処法
   6. リーダーシップ
   7. 適職・キャリア
   0. 戻る
   [1-7/0/help]:
   ```
   `0. 戻る` の遷移先は Req 5.2 に従い **遷移元状態** に戻る（TYPE_SELECT から入った場合は TYPE_SELECT へ、DIAGNOSIS_RESULT から入った場合は DIAGNOSIS_RESULT へ）。メニュー文言は遷移先に依存しない固定表記とする
3. When カテゴリ番号（1〜7）が選択される, the MBTIボット shall コンテンツを取得・表示した後 TYPE_CATEGORY状態を維持し（同じカテゴリメニューを再表示する）
4. The MBTIボット shall 16タイプ全て（ESTJ/ESTP/ESFJ/ESFP/ENTJ/ENTP/ENFJ/ENFP/ISTJ/ISTP/ISFJ/ISFP/INTJ/INTP/INFJ/INFP）に対応する
5. If 正規化後も16タイプに該当しない入力が来る, the MBTIボット shall 「無効なタイプ名です（例: INTJ）。再入力してください。」と表示して TYPE_SELECT状態を維持する

### Requirement 4: 心理機能・心理傾向軸の解説機能
**Objective:** ユーザーとして、MBTIの基礎理論を調べたい。そうすることで、タイプの背景にある心理機能の意味を理解できる。

#### 正本名テーブル（機能・軸）

The MBTIボット shall 以下の **機能正本名テーブル** および **軸正本名テーブル** を唯一の正とし、UI表示（見出し・ヘッダー）、Claude プロンプト（Req 15.2）、テンプレート見出し（Req 11.1）の全てで同じ名前を使用する。URLスラッグ（Req 7.1）は URL構築専用であり、正本名とは別物とする

**機能正本名テーブル:**

| 機能略称 | 機能フルネーム |
|---|---|
| Se | 外向的感覚 |
| Si | 内向的感覚 |
| Ne | 外向的直観 |
| Ni | 内向的直観 |
| Te | 外向的思考 |
| Ti | 内向的思考 |
| Fe | 外向的感情 |
| Fi | 内向的感情 |

**軸正本名テーブル:**

| 軸略称 | 軸フルネーム |
|---|---|
| EI | 外向と内向 |
| SN | 感覚と直観 |
| TF | 思考と感情 |
| JP | 規範と柔軟 |

#### Acceptance Criteria
1. When FUNCTION_SELECT状態で有効な機能略称が入力される, the MBTIボット shall コンテンツを取得・表示した後 FUNCTION_SELECT状態を維持し、**ヒント行（5.6の形式）のみ**再表示して入力待ちに戻る（初期表示ヘッダーの再描画は行わない）
2. When AXIS_SELECT状態で有効な軸略称が入力される, the MBTIボット shall コンテンツを取得・表示した後 AXIS_SELECT状態を維持し、**ヒント行のみ**再表示して入力待ちに戻る（初期表示ヘッダーの再描画は行わない）
3. The MBTIボット shall **テンプレートA/B** での心理機能コンテンツ表示において、本文テキスト（最大800字）の直後・AI生成ラベル（テンプレートAのみ）または出典行の直前に、以下の形式でタイプ一覧を**本文外のセクションとして常に**追加する（800字制限の対象外。Webコンテンツや API生成テキストに既に一覧が含まれるかにかかわらず、ビルトイン定数から生成して追記する）

   ```
   主/補機能タイプ一覧: [略称]: [タイプ名](主), [タイプ名](主), [タイプ名](補), ...
   ```

   例: `主/補機能タイプ一覧: Ni: INFJ(主), INTJ(主), ENFJ(補), ENTJ(補)`

   **並び順:** 主機能タイプ（`(主)`）を先に全て列挙し、次に補助機能タイプ（`(補)`）を列挙する。**各グループ内はMBTIタイプコードのアルファベット昇順**（例: ENFJ < ENTJ < INFJ < INTJ）とする

   テンプレートCはビルトインテキスト自体にタイプ一覧が含まれるため（Req 10.6参照）、追記しない
4. If FUNCTION_SELECT状態で正規化後の入力が Se/Si/Ne/Ni/Te/Ti/Fe/Fi のいずれにも一致しない, the MBTIボット shall 「無効な機能略称です（例: Ni）。再入力してください。」と表示して FUNCTION_SELECT状態を維持する
5. If AXIS_SELECT状態で正規化後の入力が EI/SN/TF/JP のいずれにも一致しない, the MBTIボット shall 「無効な軸略称です（例: SN）。再入力してください。」と表示して AXIS_SELECT状態を維持する

### Requirement 5: CLIの状態遷移（一意な定義）
**Objective:** 実装者として、状態機械を一意に実装できるよう遷移ルールを明確にしたい。そうすることで、画面ごとの入力解釈が競合しない。

#### Acceptance Criteria
1. The MBTIボット shall 以下の **7状態** で動作する状態機械を実装する

   | 状態ID | 説明 |
   |---|---|
   | MAIN_MENU | 起動直後・全機能への入口 |
   | DIAGNOSIS_QUESTION | 診断質問への回答中 |
   | DIAGNOSIS_RESULT | 診断結果の表示・確認中 |
   | TYPE_SELECT | タイプ名入力待ち |
   | TYPE_CATEGORY | タイプ別カテゴリ選択中 |
   | FUNCTION_SELECT | 心理機能略称入力待ち |
   | AXIS_SELECT | 傾向軸略称入力待ち |

2. The MBTIボット shall 各状態の受理入力・遷移先を以下の表で定義する

   | 状態 | 受理入力 | 遷移先 |
   |---|---|---|
   | MAIN_MENU | `1` | DIAGNOSIS_QUESTION（診断開始）|
   | MAIN_MENU | `2` | TYPE_SELECT |
   | MAIN_MENU | `3` | FUNCTION_SELECT |
   | MAIN_MENU | `4` | AXIS_SELECT |
   | MAIN_MENU | `0` | 終了 |
   | MAIN_MENU | `help` | MAIN_MENU（ヘルプ表示後維持）|
   | DIAGNOSIS_QUESTION | `A` / `B`（大文字小文字不問）| DIAGNOSIS_QUESTION（次の問へ）→ 16問完了でDIAGNOSIS_RESULT |
   | DIAGNOSIS_QUESTION | `q` / `quit`（大文字小文字不問）| MAIN_MENU |
   | DIAGNOSIS_QUESTION | `help` | DIAGNOSIS_QUESTION（ヘルプ表示後維持）|
   | DIAGNOSIS_RESULT | `t` | TYPE_CATEGORY（診断されたタイプの詳細へ。遷移元 = DIAGNOSIS_RESULT を記録。対象タイプ = 診断確定タイプを記録）|
   | DIAGNOSIS_RESULT | `0` | MAIN_MENU |
   | DIAGNOSIS_RESULT | `help` | DIAGNOSIS_RESULT（ヘルプ表示後維持）|
   | TYPE_SELECT | 有効なタイプ名（正規化後） | TYPE_CATEGORY（遷移元 = TYPE_SELECT を記録。対象タイプ = 正規化後のタイプ名を記録）|
   | TYPE_SELECT | `0` | MAIN_MENU |
   | TYPE_SELECT | `help` | TYPE_SELECT（ヘルプ表示後維持）|
   | TYPE_CATEGORY | `1`〜`7` | TYPE_CATEGORY（表示後維持）|
   | TYPE_CATEGORY | `0` | **遷移元状態**（TYPE_SELECT または DIAGNOSIS_RESULT）。DIAGNOSIS_RESULT が遷移元の場合は全画面を再描画する（Req 5.12 参照）|
   | TYPE_CATEGORY | `help` | TYPE_CATEGORY（ヘルプ表示後維持）|
   | FUNCTION_SELECT | 有効な機能略称（正規化後） | FUNCTION_SELECT（表示後維持）|
   | FUNCTION_SELECT | `0` | MAIN_MENU |
   | FUNCTION_SELECT | `help` | FUNCTION_SELECT（ヘルプ表示後維持）|
   | AXIS_SELECT | 有効な軸略称（正規化後） | AXIS_SELECT（表示後維持）|
   | AXIS_SELECT | `0` | MAIN_MENU |

### Requirement 6: 入力の正規化ルール
**Objective:** 実装者として、大文字小文字の揺れを吸収して一意な処理に落としたい。そうすることで、ユーザーが intj や ni と入力しても正しく動作する。

#### Acceptance Criteria
1. The MBTIボット shall ユーザーの全テキスト入力に対して **前後の空白をトリム** してから処理する。トリム対象には全角スペース（U+3000, `　`）を含める（全角スペースを半角スペースに変換した後に前後トリムする）
2. The MBTIボット shall トリムの後、**全角英字（Ａ〜Ｚ、ａ〜ｚ）および全角数字（０〜９）をそれぞれ対応する半角英字・数字に変換**してから 6.3 以降の各ルールを適用する（例: ＩＮＴＪ → INTJ、Ｎｉ → Ni、０ → 0、１ → 1）
3. The MBTIボット shall 6.1〜6.2 の共通正規化（トリム・全角半角変換）の後、**以下の順序で判定する**。状態別の形式変換（大文字化等）は予約コマンドに一致しなかった場合にのみ適用する

   **判定順序（全状態共通）:**
   1. トリム・全角半角変換後の文字列を **小文字化** して `help` と比較 → 一致すれば help コマンドとして処理（6.4〜6.9 の状態別形式変換は適用しない）
   2. トリム・全角半角変換後の文字列が `0` と一致するか確認 → 一致した場合、**現在の状態が DIAGNOSIS_QUESTION であれば無効入力として処理**（Req 5.5。`0` コマンドとしては扱わない）、**それ以外の状態であれば `0` コマンドとして処理**
   3. **状態固有の予約コマンド** を判定（下記参照）
   4. 上記いずれにも一致しなければ、**状態別の形式変換**（6.4〜6.9）を適用してからドメイン値（タイプ名・機能略称・軸略称等）として判定

   **状態固有の予約コマンド（判定順序ステップ3）:**
   - MAIN_MENU: `1`・`2`・`3`・`4` → 各メニュー項目へ遷移（Req 5.2 参照）
   - DIAGNOSIS_QUESTION: 大文字化して `Q` または `QUIT` → 中断。大文字化して `A` または `B` → 回答（全て大文字化で統一。入力 `q`/`Quit`/`QUIT` はいずれも大文字化後 `Q`/`QUIT` に一致する）
   - DIAGNOSIS_RESULT: 小文字化して `t` → TYPE_CATEGORY へ遷移
   - TYPE_CATEGORY: `1`〜`7` → カテゴリ表示（Req 5.2 参照）

   数値コマンド `1`〜`7` は 6.2 の全角半角変換後の文字列に対して判定する（例: `１` → `1`、`７` → `7`）。MAIN_MENU での `1`〜`4` と TYPE_CATEGORY での `1`〜`7` はそれぞれの状態でのみ有効であり、他の状態では無効入力として処理される

   **注意:** DIAGNOSIS_QUESTION での `0` は判定順序ステップ2 で明示的に無効入力として処理される。診断中断は `q`/`quit`（ステップ3）のみ

4. The MBTIボット shall タイプ名入力（TYPE_SELECT）の状態別形式変換として **大文字に変換** して判定する（例: `intj` → `INTJ`）。この変換は 6.3 の判定順序ステップ4でのみ適用する
5. The MBTIボット shall 心理機能入力（FUNCTION_SELECT）の状態別形式変換として **先頭大文字・2文字目小文字** に変換して判定する（例: `SE` → `Se`、`se` → `Se`）。この変換は 6.3 の判定順序ステップ4でのみ適用する
6. The MBTIボット shall 傾向軸入力（AXIS_SELECT）の状態別形式変換として **全文字を大文字に変換** して判定する（例: `ei` → `EI`）。入力文字数の制限は行わない（`help` は 6.3 のステップ1で先に判定されるため衝突しない。`HELP` を大文字化しても軸略称 EI/SN/TF/JP に一致しないため無効入力として処理される）。この変換は 6.3 の判定順序ステップ4でのみ適用する
7. The MBTIボット shall 診断回答（DIAGNOSIS_QUESTION）の `A`/`B`/`Q`/`QUIT` を 6.3 の判定順序ステップ3で **大文字変換** した後に判定する（`A`/`B`/`Q`/`QUIT` 全て大文字化で統一。6.3 ステップ3 の記述と一致）
8. The MBTIボット shall `help` の判定を 6.3 のステップ1で大文字小文字不問で行う（`HELP` / `Help` / `ｈｅｌｐ` も受理）
9. The MBTIボット shall DIAGNOSIS_RESULT状態の `t` 入力を 6.3 のステップ3で大文字小文字不問で判定する（`T` も受理）

### Requirement 8: Webコンテンツ取得・整形・コンテキスト制御
**Objective:** 実装者として、WikiのHTMLからテキストを安全に抽出し、読みやすく整形した上でAPIのトークン制限内に収めたい。そうすることで、コスト超過や出力の崩れを防げる。

#### Acceptance Criteria
1. When WikiへのHTTPリクエストが **HTTP 200** を返す, the MBTIボット shall HTMLからメインコンテンツのテキストのみを抽出する。HTMLパーサには **`cheerio`**（npmパッケージ）を使用する（唯一の許可HTMLパーサ依存。CSSセレクタによるDOM操作を安全に行うため、正規表現ベースの独自パーサは禁止する）。以下のルールに従う:
   - 抽出対象: `#content` 要素内のテキスト
   - 抽出前に除去する要素: `script`、`style`、`#menubar`、`#edit-menu`、`#responsive-navigation`、`#footer`、`#share-button-root`、`#admin-contact-root`、`id` 属性が `div-gpt-ad-` で始まる要素（cheerioセレクタ: `[id^="div-gpt-ad-"]`）、**クラス名トークン**が `ad-` で始まる要素。実装は cheerio の `.filter()` で各要素の `class` 属性を検査する。**`class` 属性が存在しない要素はスキップする**（除去しない）。`class` 属性が存在する場合、`(element.attribs.class || '').split(/\s+/).filter(Boolean)` でトークン分割し、いずれかのトークンが `ad-` で始まるかを判定する（`token.startsWith('ad-')`）。CSSセレクタ `[class^="ad-"], [class*=" ad-"]` は近似だがタブ区切り等のエッジケースを見落とすため、プログラム判定を正とする。例: `class="sidebar ad-banner"` → 除去対象、`class="loading-badge"` → 除去しない、`class` 属性なし → 除去しない、`class=""` → 除去しない
   - `#content` 要素が存在しない場合はテキスト0文字として扱い 8.2 のフォールバックへ進む
2. The MBTIボット shall 抽出したテキストが **0文字の場合** はHTTP 200でも取得失敗として扱い、Req 9 のフォールバック処理へ進む
3. The MBTIボット shall 8.1 の除去後の `#content` 要素に対して、以下の **固定アルゴリズム** でテキストを抽出・整形する。DOMを破壊的に変更する方式（replaceWith等）は使用しない（親子関係の破壊で結果が不安定になるため）。代わりに **再帰走査** でテキストを組み立てる

   **アルゴリズム（擬似コード）:**
   ```
   BLOCK_TAGS = new Set(['h1','h2','h3','h4','h5','h6','p','div','li','tr','br'])
   CELL_TAGS  = new Set(['td','th'])

   function extractNode(node, $):
     // テキストノード
     if node.type == 'text':
       return node.data

     // 要素ノード以外（コメント等）はスキップ
     if node.type != 'tag':
       return ''

     tag = node.name.toLowerCase()

     // a タグ — 子のテキストのみ（href は含めない）
     // 子を再帰走査して結合
     childText = ''
     for each child in node.children:
       childText += extractNode(child, $)

     // br — 改行に置換（子は持たないが安全のため）
     if tag == 'br':
       return '\n'

     // td/th — 子テキストの後にスペースを付加
     if CELL_TAGS.has(tag):
       return childText + ' '

     // ブロック要素 — 子テキストの後に改行を付加
     if BLOCK_TAGS.has(tag):
       return childText + '\n'

     // その他のインライン要素 — 子テキストをそのまま返す
     return childText

   function extractText($content):
     rawText = ''
     for each child in $content[0].children:
       rawText += extractNode(child, $)

     // 後処理
     lines = rawText.split('\n')
     lines = lines.map(line => line.trim())       // 各行トリム
     text = lines.join('\n')
     text = text.replace(/\n{3,}/g, '\n\n')       // 3行以上の連続改行を2行に圧縮
     text = text.trim()                            // 先頭末尾の空白・改行を除去
     return text
   ```

   **注意事項:**
   - DOMの破壊的変更（replaceWith, remove, html() 書き換え等）は `extractText` 内で行わないこと。8.1 の除去は `extractText` 呼び出し前に完了している前提
   - 再帰走査は `#content` の直接の子から開始し、深さ優先で全ノードを訪問する
   - 整形後のテキストが **0文字** の場合は取得失敗（8.2と同様）として扱う
4. The MBTIボット shall 整形後のテキストが **5,000文字を超える場合**、先頭から5,000文字で切り詰めてClaude APIのコンテキストとして使用する（先頭を優先するのは、Wikiページの重要な説明が冒頭に集中しているため）
5. The MBTIボット shall 表示用テキスト（テンプレートB）を整形後テキストから先頭800文字で切り出す。切り出し位置が文の途中（直前に `。`・`！`・`？`・`\n` が存在しない）かつ800文字から遡って50文字以内に文末記号があれば、その位置まで戻して切り詰める。それでも見つからない場合は800文字でそのまま切る。**文字数は JavaScript の `String.length`（UTF-16コード単位）で計測する**（ASCII・ひらがな・漢字等の日本語文字はいずれも1カウント。絵文字等の UTF-16サロゲートペアは2カウントになるが、本仕様の対象テキストには含まれないため考慮しない）
6. The MBTIボット shall 同一セッション内で **取得成功（整形後テキスト1文字以上）した** URLへの2回目以降のリクエストをメモリキャッシュ（Map）から返し、HTTPリクエストを送信しない。キャッシュ対象は **整形済み抽出テキストのみ**とし、Claude API 要約結果はキャッシュしない。キャッシュヒット時でも APIモードであれば Claude API を毎回呼び出す（テンプレート選択は 11.3 のロジックに従う。ヒット後に 401 が返った場合もモード切り替えを行う）。失敗結果（エラー・タイムアウト・テキスト0文字）はキャッシュしない（次回リクエスト時に再試行する）
7. The MBTIボット shall `AbortSignal.timeout(10000)` を使用して、HTTPリクエスト全体（接続確立からレスポンス本文取得完了まで）に **10秒** のタイムアウトを設定する。接続成功後の本文受信中にタイムアウトした場合も `AbortError` として Req 9.1 の優先順位2で処理する
8. The MBTIボット shall HTTPリダイレクトは `fetch` の自動追従（デフォルト動作）に任せ、リダイレクト先URLの追跡・ログ出力は行わない。キャッシュキーは **Req 7.1 で生成した元URL**（リダイレクト前のURL）とする
9. The MBTIボット shall wikiwiki.jp へのHTTPリクエストにおいて以下の運用条件を遵守する
   - **User-Agent**: `MBTIBot/1.0 (Node.js; educational-use)` を固定で設定する
   - **同時接続数**: wikiwiki.jp への同時HTTPリクエストは **1件** とする（CLI は逐次操作であり並列取得は不要。将来の並列化時にも同時1件を維持すること）
   - **リクエスト間隔**: 明示的なスロットリングは不要（ユーザーの手動操作間隔が自然なレートリミットとなるため）。ただし自動化されたバッチ取得・プリフェッチは行わない
   - **サイト側の HTTP 429/403**: Req 9.1 の優先順位4（その他のHTTPエラー）として処理する。特別な再試行やバックオフは行わない

### Requirement 9: 通信エラー処理（優先順位と再試行）
**Objective:** 実装者として、通信エラーの種別ごとに一意な処理フローを持ちたい。そうすることで、ユーザーへの表示とフォールバック先を一意に決定できる。

#### Acceptance Criteria
1. The MBTIボット shall HTTPリクエストのエラー処理を以下の優先順位で行う

   | 優先順位 | 条件 | 再試行 | ユーザーへの表示 | その後の処理 |
   |---|---|---|---|---|
   | 1 | HTTP 404 | **しない**（即確定） | 「このページはwikiwiki.jpに存在しません。」 | Req 10 ビルトイン表示 |
   | 2 | タイムアウト（AbortController による `AbortError`）/ 接続エラー（DNS失敗・接続拒否・TLS失敗等による `TypeError`） | **1回のみ**再試行、再試行後も失敗なら確定。**再試行が成功（HTTP 200 かつテキスト1文字以上）した場合は完全に通常フローへ戻る**（エラーメッセージは表示せず、テンプレート選択は Req 11.3 に従う） | 初回:（非表示）/ 再試行後失敗: 再試行後の結果エラー種別を優先順位1/2/3/4のいずれかで判定してその文言を表示（例: 2回目が404なら「このページはwikiwiki.jpに存在しません。」、2回目もタイムアウト/接続エラーなら「Webコンテンツの取得に失敗しました。」） | 再試行失敗時: Req 10 ビルトイン表示 |
   | 3 | HTTP 200 だがテキスト0文字 | **しない**（即確定） | 「ページから情報を抽出できませんでした。」 | Req 10 ビルトイン表示 |
   | 4 | その他のHTTPエラー（4xx/5xx） | **しない**（即確定） | 「Webコンテンツの取得に失敗しました（HTTPステータス: [コード]）。」 | Req 10 ビルトイン表示 |

2. The MBTIボット shall Claude APIのエラー処理を以下の優先順位で行う（再試行は行わない）。Claude API呼び出しはWeb取得成功後にのみ行われるため、この表の全ケースにおいてWeb取得済みテキストは利用可能である

   | 優先順位 | 条件 | ユーザーへの表示 | その後の処理 |
   |---|---|---|---|
   | 1 | HTTP 401 | 「APIキーが無効です。ANTHROPIC_API_KEY を確認してください。」 | **プロセス終了まで**スタンドアロンモードへ切り替え（1.3参照）、**テンプレートBで表示**（Web取得済みテキストを使用） |
   | 2 | HTTP 429 | 「APIのレート制限に達しました。しばらく待ってから再試行してください。」 | **テンプレートBで表示**（Web取得済みテキストを使用） |
   | 3 | タイムアウト（30秒） | 「APIの応答がタイムアウトしました。」 | **テンプレートBで表示**（Web取得済みテキストを使用） |
   | 4a | **401・429以外の非2xxレスポンス** | 「API呼び出しに失敗しました（HTTP [ステータスコード]）。」（例: `API呼び出しに失敗しました（HTTP 500）。`） | **テンプレートBで表示**（Web取得済みテキストを使用） |
   | 4b | **ネットワーク例外**（AbortError以外のTypeError等） | 「API呼び出しに失敗しました（ネットワークエラー）。」 | **テンプレートBで表示**（Web取得済みテキストを使用） |
   | 5 | **HTTP 200** だがレスポンス本文が空・先頭要素の `type !== 'text'`・JSON パース失敗（優先順位4より後に評価する） | 「API呼び出しに失敗しました（レスポンスを解析できませんでした）。」 | **テンプレートBで表示**（Web取得済みテキストを使用） |

   **優先順位4a/4bの制約:** 上記2パターンの固定文言のみを許可する。生の例外メッセージ・スタックトレース・内部エラーオブジェクトの文字列化をエラーメッセージに含めてはならない（Req 13.6 との整合）

3. The MBTIボット shall エラーメッセージとその後のフォールバックテンプレートを以下の **固定レイアウト** で出力する（Req 9.1 / 9.2 の全ケースに適用）
   ```
   [エラーメッセージ（1行）]

   === [見出し内容] ===
   [テンプレートB または テンプレートC の本文]
   ...
   ```
   エラーメッセージとテンプレート見出しの間は **空行1行のみ**を挿入する。エラーメッセージは常にテンプレート本文の **前** に出力する

4. The MBTIボット shall エラーフォールバック表示（テンプレートB/C）の **後** に、各状態の通常時と同じ再表示ルールを適用する

   | 状態 | フォールバック表示後の再表示 |
   |---|---|
   | TYPE_CATEGORY | カテゴリメニューを再表示する（Req 3.3 と同じ） |
   | FUNCTION_SELECT | ヒント行のみ再表示する（Req 4.1 と同じ） |
   | AXIS_SELECT | ヒント行のみ再表示する（Req 4.2 と同じ） |

   エラーフォールバックはコンテンツ表示の一形態であり、成功時と失敗時で再表示の挙動を変えない

### Requirement 13: ログ方針
**Objective:** 実装者として、デバッグに必要な情報を取りつつプライバシーに配慮したい。そうすることで、問題発生時に原因を追えるが個人情報は残らない。

### Requirement 15: Claude API連携
**Objective:** ユーザーとして、WikiコンテンツをベースにしたAI要約を受けたい。そうすることで、静的表示を超えた詳細な説明を得られる。

#### Acceptance Criteria
1. The MBTIボット shall 使用モデルを `ANTHROPIC_MODEL` 環境変数で上書き可能にする。**実効値** は以下のルールで起動時に1回だけ決定し、バナー表示（Req 1.4）・APIリクエスト送信（Req 15.2）の両方でこの実効値を使用する:
   - `ANTHROPIC_MODEL` が未設定・空文字列・空白のみの値（トリム後に0文字）→ デフォルト値 `claude-sonnet-4-6`
   - 上記以外 → **トリム後の値** をそのまま使用する（例: `" claude-sonnet-4-6 "` → `claude-sonnet-4-6`）
2. When Claude APIを呼び出す, the MBTIボット shall Node.js native `fetch`（外部ライブラリなし）を用いて以下のHTTPリクエストを送信する

   **HTTP契約:**
   - メソッド: `POST`
   - エンドポイント: `https://api.anthropic.com/v1/messages`
   - ヘッダー: `Content-Type: application/json`、`x-api-key: [ANTHROPIC_API_KEY]`、`anthropic-version: 2023-06-01`
   - リクエストボディ（JSON）:
     ```json
     {
       "model": "[ANTHROPIC_MODEL]",
       "max_tokens": 1024,
       "system": "[システムプロンプト]",
       "messages": [{ "role": "user", "content": "[ユーザーメッセージ]" }]
     }
     ```
   - レスポンス本文の取得（以下の順序で処理する）:
     1. `response.status` が 2xx 以外 → Req 9.2 の優先順位1〜4 で処理（JSONパースは行わない）
     2. `response.status` が 2xx → `await response.json()` でレスポンスボディを JSON デコードする
     3. JSON デコード成功 → デコード結果を検証する。以下のいずれかに該当する場合は Req 9.2 の優先順位5で処理する:
        - デコード結果が **オブジェクトでない**（`json === null || typeof json !== 'object' || Array.isArray(json)`。`null`・配列・プリミティブ値はすべて不正）
        - `content` が **存在しない・配列でない**（`!Array.isArray(json.content)`）
        - `content` 配列が空（`json.content.length === 0`）
        - 先頭要素の `type` が `'text'` でない
        - 先頭要素の `text` が **文字列でない**（`typeof json.content[0].text !== 'string'`。`undefined`・`null`・数値等）
        - 先頭要素の `text` が **空文字列**（`json.content[0].text.trim().length === 0`）
     4. 上記検証を通過した場合、`json.content[0].text` をレスポンステキストとして使用する
     5. JSON デコード失敗（SyntaxError等）→ Req 9.2 の優先順位5で処理
     6. 後続要素（`json.content[1]` 以降）は参照しない

   **システムプロンプト**: 以下の固定テキストの後にWikiコンテンツ（最大5,000字）を連結する
     ```
     あなたはMBTI（Myers-Briggs Type Indicator）の専門家です。
     以下のwikiwiki.jp/16typesの内容のみを根拠として日本語で回答してください。
     提供されたテキストに記載のない情報を補足・推測・外挿してはいけません。
     回答は箇条書きまたは短い段落で構成し、簡潔にまとめてください。

     --- 参考テキスト ---
     [Wikiコンテンツ]
     ```
   **ユーザーメッセージ**: 表示コンテキストに応じて以下を使用する
     - タイプ別カテゴリ: `[タイプ名]タイプの「[カテゴリ名]」について詳しく説明してください。`（`[カテゴリ名]` は Req 3.2 のカテゴリ正本名テーブルの値を使用する）
     - 心理機能: `[機能略称]（[機能フルネーム]）について詳しく説明してください。`（例: `Ni（内向的直観）について詳しく説明してください。`）
     - 心理傾向軸: `[軸略称]（[軸フルネーム]）について詳しく説明してください。`（例: `SN（感覚と直観）について詳しく説明してください。`）
3. The MBTIボット shall Claude API呼び出し時に `max_tokens` を **1,024** に設定する（日本語800字相当のバッファ）。レスポンステキストが800文字を超える場合は 8.5 の文末記号考慮ルールと同じ方法で先頭800文字に切り詰める（テンプレートAの「最大800字」保証）
4. The MBTIボット shall Claude APIのタイムアウトを **`AbortSignal.timeout(30000)`** で設定する（Web取得の Req 8.7 と同じ方式。計測範囲は fetch 呼び出しからレスポンス本文取得完了まで。タイムアウト時は `AbortError` が発生し、Req 9.2 の優先順位3で処理する）
5. The MBTIボット shall Claude APIへの再試行は行わない（エラー時は即 Req 9 のエラー処理へ）
6. The MBTIボット shall AI要約の品質を **プロンプト仕様の遵守** によって担保する。本要件が受け入れ可能とする条件は以下の2点のみとする（セマンティックな幻覚検出は自動テストの対象外とし、実装者の責任範囲外とする）
   - 条件A: 送信するシステムプロンプトが 15.2 の固定テキスト（参考テキスト連結前）と **文字列一致** すること
   - 条件B: API から返ったレスポンステキストを **加工なしでテンプレートに埋め込む**こと（800字超時の切り詰めを除く）
