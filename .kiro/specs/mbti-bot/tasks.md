- [x] 6. Web コンテンツ取得モジュールの実装
- [ ] 6.1 HTTP 取得とキャッシュを実装する
  - native fetch + `AbortSignal.timeout(10000)` で HTTP リクエストを行う
  - User-Agent を `MBTIBot/1.0 (Node.js; educational-use)` に固定する
  - 成功テキストを `Map` でメモリキャッシュする（キーはリダイレクト前の元URL）。キャッシュ対象は**整形済み抽出テキストのみ**であり、Claude API 要約結果はキャッシュしない
  - キャッシュヒット時でも API モードであれば Claude API を毎回呼び出す（要約結果の再利用は禁止）
  - 失敗結果はキャッシュしない（次回リクエスト時に再試行する）
  - HTTPリダイレクトは fetch のデフォルト自動追従に任せる
  - wikiwiki.jp への同時接続数は1件とする（並列取得・プリフェッチ禁止）
  - HTTP リクエストの成功・失敗いずれの場合も DebugLogger を呼び出し、URL・ステータスコード・レスポンスタイム（エラー時はエラー種別・メッセージ）を記録する
  - _Requirements: 8.6, 8.7, 8.8, 8.9_

- [ ] 6.2 cheerio による HTML 抽出と不要要素除去を実装する
  - `#content` 要素内のテキストを抽出対象とする（不在時はテキスト0文字）
  - 除去対象: `script`, `style`, `#menubar`, `#edit-menu`, `#responsive-navigation`, `#footer`, `#share-button-root`, `#admin-contact-root`, `[id^="div-gpt-ad-"]`
  - クラストークン判定による広告要素除去: `class` 属性をトークン分割し、`ad-` で始まるトークンがあれば除去。`class` 属性なし・空の要素はスキップ
  - _Requirements: 8.1_

- [ ] 6.3 再帰走査によるテキスト整形を実装する
  - DOM を非破壊で再帰走査してテキストを組み立てる（`replaceWith` 等は使用禁止）
  - ブロック要素（`h1`〜`h6`, `p`, `div`, `li`, `tr`, `br`）の後に改行を付加する
  - テーブルセル（`td`/`th`）の後にスペースを付加する
  - インライン要素（`a` 含む）は子テキストをそのまま返す
  - 後処理: 各行トリム → 3行以上の連続改行を2行に圧縮（`/\n{3,}/g` → `\n\n`）→ 先頭末尾トリム
  - 整形後テキストが0文字なら取得失敗として扱う
  - _Requirements: 8.2, 8.3_

- [ ] 6.4 テキスト切り詰め処理を実装する
  - API コンテキスト用に先頭5,000文字で切り詰める
  - 表示用に先頭800文字で切り詰める。切り出し位置が文の途中なら50文字以内で `。！？\n` を探して戻す
  - 文末記号が見つからなければ800文字でそのまま切る
  - 文字数は `String.length`（UTF-16コード単位）で計測する
  - _Requirements: 8.4, 8.5_

- [ ] 6.5 Web 取得エラー処理と再試行を実装する
  - HTTP 404 → 即確定で固定メッセージ
  - タイムアウト/接続エラー → **初回はユーザーに何も表示せず**1回のみ再試行する。再試行後の結果を優先順位1〜4で再分類する
  - HTTP 200 + テキスト0文字 → 即確定で固定メッセージ
  - その他 4xx/5xx → 即確定でステータスコード付き固定メッセージ
  - 再試行が成功した場合は通常フローへ完全復帰する
  - _Requirements: 9.1_

- [x] 7. Claude API 連携モジュールの実装
- [ ] 7.1 API リクエスト送信を実装する
  - native fetch + `AbortSignal.timeout(30000)` で POST リクエストを送信する
  - エンドポイント・ヘッダー3点・ボディ構造を要件の HTTP 契約に厳密に従う
  - 固定システムプロンプトの全文を定数として保持し、Wiki コンテンツを連結する
  - ユーザーメッセージを正本名テーブルの値を使用して生成する（3パターン）
  - `max_tokens` を 1024 に設定する
  - API 呼び出しの成功・失敗いずれの場合も DebugLogger を呼び出し、モデル名・推定入力トークン数・レスポンスタイム（エラー時はステータスコード・エラー種別・メッセージ）を記録する
  - _Requirements: 15.2, 15.3, 13.3_

- [ ] 7.2 レスポンス検証とエラー処理を実装する
  - 非2xx → ステータスコードに基づく優先順位判定: 401 →「APIキーが無効です。ANTHROPIC_API_KEY を確認してください。」、429 →「APIのレート制限に達しました。しばらく待ってから再試行してください。」、その他 →「API呼び出しに失敗しました（HTTP [ステータスコード]）。」
  - タイムアウト（`AbortError`）→「APIの応答がタイムアウトしました。」
  - ネットワーク例外（`AbortError` 以外の `TypeError` 等）→「API呼び出しに失敗しました（ネットワークエラー）。」。fetch が例外を送出した場合に `AbortError` かそれ以外かで分岐する
  - 2xx → JSON デコード → ルート型チェック → content 配列チェック → content[0].type/text 検証。いずれかの検証失敗 →「API呼び出しに失敗しました（レスポンスを解析できませんでした）。」
  - 検証通過した text を加工なしで返却する（800字超の場合のみ文末考慮で切り詰め）
  - 401 時は降格シグナル（`is401: true`）を返す
  - API への再試行は行わない
  - _Requirements: 9.2, 15.4, 15.5, 15.6_
